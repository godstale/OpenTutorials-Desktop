# Windows Zip 경로 구분자 처리 방식

## 개념 개요
웹 서버, 클라우드 오브젝트 스토리지(예: Supabase Storage, AWS S3) 및 대부분의 유닉스 기반 웹 환경은 경로 구분자로 항상 슬래시(`/`)를 사용합니다. 
반면 Windows 환경에서 압축되거나 빌드된 zip 파일을 다룰 때, 압축 라이브러리(예: `AdmZip`)를 통해 내부 파일 엔트리 목록을 추출하면 OS 특성에 따라 백슬래시(`\`)가 경로 구분자로 반환될 수 있습니다 (예: `cards\01-intro.mdx`).

이러한 OS 종속적인 구분자 표현식 차이는 업로드 경로 뒤틀림, 정적 자원 404 오류 등을 유발하는 대표적인 크로스 플랫폼 호환성 장애의 원인입니다.

## 원칙 및 대응 방식

1. **경로 표준화 (Normalization)**:
   - 압축 파일 내부의 엔트리 이름(`entry.entryName`)이나 파일 시스템의 경로 정보를 클라우드 저장소 경로 또는 웹 URL에 그대로 대입하지 않아야 합니다.
   - 업로드 또는 파일 처리를 실행하기 전에 경로 문자열 내의 모든 백슬래시(`\`)를 슬래시(`/`)로 치환해 주는 표준화가 필수적입니다.
   ```typescript
   const safePath = entryName.replace(/\\/g, '/');
   ```

2. **Supabase Storage 등의 버킷 저장 구조 규칙**:
   - 클라우드 버킷은 내부적으로 디렉토리 개념 대신 단일 Key 문자열을 사용합니다.
   - 따라서 `cards\01-intro.mdx` 형태로 키가 생성되어 저장되면, 조회 API 및 CDN 엣지에서는 슬래시 기반 경로(`/cards/01-intro.mdx`)로 파일을 탐색하므로 자원을 조회할 수 없게 됩니다.
   - 업로드 단계에서 반드시 슬래시로 정규화하여 경로 구조를 일치시킵니다.

3. **크로스 플랫폼 테스트 환경**:
   - 윈도우 로컬 머신에서 zip 번들을 압축하여 업로드 테스트를 함으로써 이와 같은 엣지 케이스를 사전에 탐색하고 조치할 수 있도록 업로드 API 파이프라인에서 정규화 처리가 항상 기본으로 탑재되어야 합니다.
